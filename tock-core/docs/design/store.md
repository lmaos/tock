# Store 模块设计

`store` 目录保存延时执行对象，属于调度与 Worker 之间的中间层。

## 核心类

| 类 | 作用 |
| --- | --- |
| `JobExecution` | 待执行任务载荷 |
| `JobStore` | 存储接口 |
| `MemoryJobStore` | 本地实现 |

## 设计说明

- 这里存的是“将来要执行的计划”，不是已经执行的结果
- 当前默认链路里它是保留接口，但仍然适合自定义调度器接入
- `MemoryJobStore` 主要服务测试和单机场景

## 依赖链路

- `JobExecution` 是存储的基本对象
- `Tock` 和 `Config` 保留了 `jobStore` 注入点
- 现有的 `EventDrivenCronScheduler` / `DefaultTockWorker` 主链路没有直接使用它

## 实现思路

这个模块更像是“保留扩展点”，不是当前默认路径的核心数据结构。  
它保留在公共 API 里，是为了以后接入其他调度策略时不需要重新设计延时任务抽象。

## 为什么这样设计

- **兼容未来扩展**：不把延时任务能力锁死在当前调度器里
- **保留接口边界**：即使暂时不用，也不影响已有 API 结构
- **测试友好**：内存实现可以配合不同调度器做实验
